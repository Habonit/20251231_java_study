# Project 문서 가이드

이 디렉토리는 프로젝트의 **큰 틀**을 정의하는 개념적/아키텍처 문서를 관리한다.

---

## 1. 개념

### 1.1 project/ vs sdd/

| 디렉토리 | 역할 | 특징 |
|:--------|:-----|:-----|
| `docs/project/` | **큰 틀** - 프로젝트 전체 개념, 아키텍처 | 변경 빈도 낮음 |
| `docs/sdd/` | **작은 단위** - 개별 작업 스펙 | 매 개발 사이클 변경 |

```
docs/project/                     ← 큰 틀 (상위 레벨)
    architecture.md
    domain_model.md
           │
           │ 참조
           ▼
docs/sdd/{브랜치 식별자}/          ← 작은 단위 (하위 레벨)
    feature_spec.md
    report.md
```

### 1.2 문서 역할

- `sdd/`의 개별 작업이 참조하는 상위 레벨 문서
- 프로젝트 전체의 방향성과 구조를 정의
- 새로운 팀원 온보딩 시 참고 자료

---

## 2. 문서 유형

| 문서 유형 | 설명 | 예시 |
|:---------|:-----|:-----|
| 아키텍처 문서 | 시스템 구조, 컴포넌트 관계 | `architecture.md` |
| 도메인 모델 | 핵심 도메인 개념 정의 | `domain_model.md` |
| 기술 스택 | 사용 기술 및 선택 이유 | `tech_stack.md` |
| 설계 결정 | 주요 기술적 결정 기록 (ADR) | `adr/001_database_choice.md` |
| 용어 사전 | 프로젝트 용어 정의 | `glossary.md` |

## 3. 문서 작성 가이드

### 3.1 작성 시점

| 시점 | 작성 문서 |
|:-----|:---------|
| 프로젝트 시작 | 아키텍처, 도메인 모델, 기술 스택 |
| 주요 기술 결정 시 | 설계 결정 기록 (ADR) |
| 새로운 도메인 추가 시 | 도메인 모델 업데이트 |

### 3.2 작성 원칙

1. **안정적인 내용**: 자주 변경되지 않는 핵심 개념만 포함
2. **참조 용이성**: `sdd/`에서 쉽게 참조할 수 있도록 명확하게 작성
3. **간결함**: 불필요한 세부사항은 `sdd/`에 위임

### 3.3 작성 참조

| 항목 | 참조 문서 |
|:-----|:-----|
| 문서 스타일 | `core/style.md` |
| 프로젝트 규칙 | `core/constitution.md` |

---

## 4. sdd/와의 관계

```
[1] project/ 문서 작성/업데이트
    (아키텍처, 도메인 모델 등)
              │
              │ 참조
              ▼
[2] sdd/{브랜치 식별자}/ 스펙 작성
    └── project/의 큰 틀을 기반으로 구체적인 구현 스펙 작성
              │
              ▼
[3] 개발 진행
              │
              ▼
[4] 필요 시 project/ 업데이트
    └── 아키텍처 변경, 새로운 도메인 추가 등
```
